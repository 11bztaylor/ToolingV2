How ADX Storage Actually Works ▼
Here's the thing that trips people up: ALL your data always lives in cold storage (Azure blob). That's the baseline. Cold storage is cheap — pennies per GB. The question is whether a copy of that data also lives somewhere faster.
Data lifecycle — what happens to a log after it's ingested
INGESTED Data arrives → stored in hot cache (SSD) AND cold storage (blob)
│ Your analysts query this data in seconds — it's on fast SSD right on the cluster nodes
CACHE EXPIRES Hot cache copy evicted → data now cold-only. Still queryable, just slower (pulls from blob)
│ Cold queries still work — all operators, all filters, full KQL. Just slower: think 10-30s vs 1-3s. For a targeted forensic query that's fine. For a dashboard refreshing every 60 seconds, it's not.
RETENTION EXPIRES Data deleted from everywhere → gone. Can't query it, can't recover it (unless recoverability is on)
Retention — "how long does data exist?"
This is your compliance knob. Legal says keep security logs for 1 year? Set retention to 365d. Need 2 years for audit? 730d. Once this expires the data is permanently deleted. Staging tables get short retention (7-30d) because data flows to typed tables almost immediately — no reason to keep the raw copy long.
Hot Cache — "how long are queries fast?"
This is your cost vs speed knob — and it's the biggest one you have. Hot cache keeps a copy of data on SSDs bolted to the cluster nodes. More hot cache = bigger SSDs needed = bigger VMs = bigger bill. For SOC work, you usually want the last 14-30 days hot (active investigation window). Older data for compliance? Let it go cold — your analysts aren't querying 8-month-old Netskope logs every day. Cache must be ≤ retention (can't cache data that doesn't exist).
Searchability impact: Hot and cold data are both fully searchable — same KQL, same operators, same results. The difference is speed. Hot queries scan data already on local SSD (1-3s). Cold queries pull data from Azure blob first, then scan it (10-30s for a focused time range, longer for broad scans). If you have dashboards, alerts, or scheduled queries hitting a table regularly — keep that window hot. If it's just occasional forensic lookups, cold is fine.
Ingestion Batching — "how fast does data show up?"
ADX doesn't write data one row at a time — it batches it up and writes in chunks. By default it waits up to 5 minutes (or 1 GB, or 1000 items — whichever comes first). That means there's up to a 5-minute gap between "event happened" and "data is queryable." For most event tables that's fine. For alerts where your SOC is actively triaging? Drop it to 30 seconds. Just know: smaller batches = more cluster work = higher cost.
Example — Netskope alert table with typical SOC settings
Retention = 365d, Hot Cache = 30d, Batching = 00:00:30
→ Alert data shows up within 30 seconds of ingestion. Analysts can query the last 30 days instantly. Older alerts (31-365 days) are still queryable for investigations/compliance but take a few seconds longer. After 1 year, data is deleted.
Format: Use d for days, h for hours. No month shorthand — 6 months = 180d, 1 year = 365d. Batching uses HH:MM:SS.
