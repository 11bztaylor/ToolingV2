How ADX Storage Actually Works ▼
Here's the thing that trips people up: ALL your data always lives in cold storage (Azure blob). That's the baseline. Cold storage is cheap — pennies per GB. The question is whether a copy of that data also lives somewhere faster.
Data lifecycle — what happens to a log after it's ingested
INGESTED Data arrives → stored in hot cache (SSD) AND cold storage (blob)
│ Your analysts query this data in seconds — it's on fast SSD right on the cluster nodes
CACHE EXPIRES Hot cache copy evicted → data now cold-only. Still queryable, just slower (pulls from blob)
│ Cold queries still work — same KQL, same results. Just slower: 10-30s vs 1-3s
RETENTION EXPIRES Data deleted from everywhere → gone. Can't query it, can't recover it (unless recoverability is on)
Retention — "how long does data exist?"
This is your compliance knob. Legal says keep security logs for 1 year? Set retention to 365d. Once this expires the data is permanently deleted. Staging tables get short retention (7-30d) because data flows to typed tables almost immediately.
Hot Cache — "how long are queries fast?"
This is your cost vs speed knob. Hot cache keeps data on SSDs bolted to cluster nodes. More hot cache = bigger VMs = bigger bill. For SOC work, keep 14-30 days hot. Older data for compliance? Let it go cold. Cache must be ≤ retention.
Ingestion Batching — "how fast does data show up?"
ADX batches writes — default waits up to 5 minutes. For alerts where SOC is actively triaging, drop to 30 seconds. Smaller batches = more cluster work = higher cost.
Format: Use d for days, h for hours. No month shorthand — 6 months = 180d, 1 year = 365d. Batching uses HH:MM:SS.
